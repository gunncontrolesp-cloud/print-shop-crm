<objective>
## Goal
Every data table is scoped to a tenant — shops can never see each other's data. All six INSERT server actions include `tenant_id`, all RLS policies enforce tenant isolation, and middleware redirects staff without a tenant to `/onboarding`.

## Purpose
This is the architectural foundation for everything in Phase 6. Without tenant isolation, every shop on the SaaS platform would see every other shop's customers, orders, and invoices. The migration converts all single-tenant tables to multi-tenant by adding `tenant_id` columns and rebuilding every RLS policy. No UI changes yet — 06-02 adds the onboarding flow that creates tenants.

## Output
- `supabase/migrations/20260327000007_multi_tenant.sql` — tenants table, tenant_id on all data tables, `get_my_tenant_id()` SECURITY DEFINER function, rebuilt RLS policies, updated `handle_new_user` trigger
- `src/lib/tenant.ts` — `getTenantId()` helper for server actions
- 6 server action files updated — all INSERT operations include `tenant_id`
- `src/lib/supabase/middleware.ts` — staff without tenant_id redirected to `/onboarding`
</objective>

    import { createClient } from '@/lib/supabase/server'

    export async function getTenantId(): Promise<string> {
      const supabase = await createClient()
      const {
        data: { user },
      } = await supabase.auth.getUser()
      if (!user) throw new Error('Not authenticated')

      const { data } = await supabase
        .from('users')
        .select('tenant_id')
        .eq('id', user.id)
        .single()

      if (!data?.tenant_id) throw new Error('No tenant assigned — complete onboarding first')
      return data.tenant_id as string
    }
