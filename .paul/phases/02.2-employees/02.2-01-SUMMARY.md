# Phase 2.2 Plan 01: Employee List + Edit Summary

**Admin can view all employees in their tenant and edit names/roles via /dashboard/settings/employees — deployed and verified in production.**

## Performance

| Metric | Value |
|--------|-------|
| Duration | ~1 session |
| Tasks | 3 auto + 1 checkpoint |
| Files modified | 4 |

## Acceptance Criteria Results

| Criterion | Status | Notes |
|-----------|--------|-------|
| AC-1: Employee list visible to admin | Pass | Both accounts visible; name falls back to email |
| AC-2: Non-admin cannot access | Pass | Redirects to /dashboard |
| AC-3: Admin can edit employee name | Pass | Verified in production |
| AC-4: Admin can change employee role | Pass | staff ↔ admin dropdown works |
| AC-5: Cannot remove own admin role | Pass | Role select disabled + hidden input preserves value |

## Accomplishments

- Employee list page live at /dashboard/settings/employees with role badges and edit links
- Edit page with name field + role dropdown, self-demotion guard
- updateEmployee server action with tenant isolation via service client
- Employees link added to nav sidebar

## Files Created/Modified

| File | Change | Purpose |
|------|--------|---------|
| `src/lib/actions/employees.ts` | Created | updateEmployee server action |
| `src/app/dashboard/settings/employees/page.tsx` | Created | Employee list (admin only) |
| `src/app/dashboard/settings/employees/[id]/edit/page.tsx` | Created | Edit employee name + role |
| `src/components/nav-sidebar.tsx` | Modified | Added Employees nav link |

## Decisions Made

| Decision | Rationale | Impact |
|----------|-----------|--------|
| Service role client for UPDATE | No UPDATE RLS policy on public.users | Future employee writes must also use service client with tenant_id guard |
| Disabled select + hidden input for self-edit | HTML disabled inputs don't submit; need hidden input to carry role value | Consistent pattern for all future role-restricted forms |

## Deviations from Plan

| Type | Count | Impact |
|------|-------|--------|
| Auto-fixed | 1 | Used service client for UPDATE — no policy gap in plan, discovered during execution |

**Auto-fix:** Plan said "use authenticated client — admin RLS policy covers this" but `public.users` has no UPDATE policy. Switched to `createServiceClient()` with explicit `.eq('tenant_id', tenantId)` guard. No functional impact.

## Issues Encountered

| Issue | Resolution |
|-------|------------|
| No UPDATE RLS policy on public.users | Used service role client with tenant_id constraint |
