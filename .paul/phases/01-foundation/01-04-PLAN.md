Build the Order module: orders table with a 6-stage status workflow (pending → approved → printing → finishing → completed → delivered), Quote → Order 1-click conversion, and order list/detail pages with status-update actions.

Orders are the production unit — once a quote is approved, it becomes an order that the production team tracks through to delivery. This plan closes the quote-to-production loop and enables the shop to stop tracking jobs on paper.

## Output
- `orders` table with status enum and RLS
- Quote → Order conversion button on approved quote detail page
- Orders list page with status-filtered view
- Order detail page with full status workflow controls
- Orders nav item enabled in sidebar
</objective>

  <action>
    Create `supabase/migrations/20260326000004_orders.sql`:

    ```sql
    create table public.orders (
      id uuid primary key default uuid_generate_v4(),
      quote_id uuid references public.quotes(id) on delete set null,
      customer_id uuid references public.customers(id) on delete cascade not null,
      status text not null default 'pending'
        check (status in ('pending', 'approved', 'printing', 'finishing', 'completed', 'delivered')),
      line_items jsonb not null default '[]',
      total numeric(10,2) not null default 0,
      notes text,
      created_by uuid references auth.users(id),
      created_at timestamptz default now(),
      updated_at timestamptz default now()
    );

    alter table public.orders enable row level security;

    -- All authenticated users can view orders
    create policy "Authenticated users can view orders"
      on public.orders for select
      using (auth.uid() in (select id from public.users));

    -- Staff + admin can create orders
    create policy "Staff can create orders"
      on public.orders for insert
      with check (auth.uid() in (select id from public.users));

    -- Staff + admin can update orders (status transitions controlled in app layer)
    create policy "Staff can update orders"
      on public.orders for update
      using (auth.uid() in (select id from public.users))
      with check (auth.uid() in (select id from public.users));

    -- Only admin can delete orders
    create policy "Admins can delete orders"
      on public.orders for delete
      using (exists (select 1 from public.users where id = auth.uid() and role = 'admin'));

    create trigger orders_updated_at
      before update on public.orders
      for each row execute procedure public.set_updated_at();
    ```

    Note: `pending → approved` role enforcement is handled at the app layer (not RLS) because RLS cannot inspect the transition direction — only the final state.

    Avoid: Adding a status enum type at the DB level — the check constraint is simpler and easier to extend without migrations.

    Status sequence for validation:
    ```ts
    const STATUS_SEQUENCE = ['pending', 'approved', 'printing', 'finishing', 'completed', 'delivered'] as const
    export type OrderStatus = typeof STATUS_SEQUENCE[number]

    function getNextStatus(current: OrderStatus): OrderStatus | null {
      const idx = STATUS_SEQUENCE.indexOf(current)
      return idx < STATUS_SEQUENCE.length - 1 ? STATUS_SEQUENCE[idx + 1] : null
    }
    ```
