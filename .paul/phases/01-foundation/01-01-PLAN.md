    ```ts
        process.env.NEXT_PUBLIC_SUPABASE_URL!,
    ```

    Create server client at `src/lib/supabase/server.ts` using `createServerClient` from `@supabase/ssr` with cookie handling (follow Supabase Next.js SSR docs pattern exactly).

    Create initial migration at `supabase/migrations/20260326000001_init_schema.sql`:
    ```sql
    -- Enable UUID extension
    create extension if not exists "uuid-ossp";

    -- Role enum
    create type user_role as enum ('admin', 'staff');

    -- Users table (extends Supabase auth.users)
    create table public.users (
      id uuid references auth.users(id) on delete cascade primary key,
      email text not null,
      name text,
      role user_role not null default 'staff',
      created_at timestamptz default now()
    );

    -- RLS: users can only read their own record; admin can read all
    alter table public.users enable row level security;

    create policy "Users can view own record"
      on public.users for select
      using (auth.uid() = id);

    create policy "Admins can view all users"
      on public.users for select
      using (
        exists (
          select 1 from public.users
          where id = auth.uid() and role = 'admin'
        )
      );

    -- Auto-insert user record on auth signup
    create or replace function public.handle_new_user()
    returns trigger as $$
    begin
      insert into public.users (id, email, role)
      values (new.id, new.email, 'staff');
      return new;
    end;
    $$ language plpgsql security definer;

    create trigger on_auth_user_created
      after insert on auth.users
      for each row execute procedure public.handle_new_user();
    ```

    Create `.env.local.example`:

    ```
    NEXT_PUBLIC_SUPABASE_ANON_KEY=your-supabase-anon-key

    ```

    Initialize Supabase local dev: `npx supabase init` then `npx supabase start`.

    Avoid: Using the service role key in client-side code. Never expose service role key to the browser.

    Create `src/lib/supabase/middleware.ts` following Supabase's official Next.js middleware pattern — refreshes session and updates cookies on every request.

    Create `src/middleware.ts`:
    ```ts
    import { type NextRequest } from 'next/server'
    import { updateSession } from '@/lib/supabase/middleware'

    export async function middleware(request: NextRequest) {
      return await updateSession(request)
    }

    export const config = {
      matcher: [
        '/((?!_next/static|_next/image|favicon.ico|.*\\.(?:svg|png|jpg|jpeg|gif|webp)$).*)',
      ],
    }
    ```

    The `updateSession` function should:
    - Refresh the Supabase session
    - If user is not authenticated AND path starts with `/dashboard`, redirect to `/login`
    - If user is authenticated AND path is `/login`, redirect to `/dashboard`

    Create `src/app/login/page.tsx`:
    - Simple email/password form using shadcn/ui Card, Input, Button
    - On submit: call `supabase.auth.signInWithPassword({ email, password })`
    - On success: `router.push('/dashboard')`
    - On error: display error message below form
    - Include a "Sign up" link that calls `supabase.auth.signUp` (admin manually promotes role after signup)

    Create `src/app/dashboard/page.tsx`:
    - Protected by middleware (unauthenticated users redirected to /login)
    - Display: "Welcome to Print Shop CRM" + user email + user role
    - Include a sign-out button: `supabase.auth.signOut()` then redirect to `/login`
    - Read user role from `public.users` table using server client

    Avoid: Storing role in JWT claims at this stage — read from `public.users` table on each dashboard load. Role promotion is manual via Supabase dashboard for now.
